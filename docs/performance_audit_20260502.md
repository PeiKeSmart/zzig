# 性能审计与语义分析报告

**日期：** 2026-05-02  
**范围：** 当前工作区相对今日开始前的 Zig 0.16 迁移与收尾改动  
**目标：**
- 检查业务逻辑和外部可见语义是否发生非预期变化
- 识别兼容改动中的性能回退点
- 在低风险前提下完成可直接落地的小优化

---

## 1. 结论摘要

### 1.1 语义结论

基于当前差异面、代码审计以及构建/测试/示例验证，**未发现业务逻辑被意外改写的证据**。

当前可确认的外部可见变化有且仅有以下几类：

1. Zig 0.16 下，完整示例构建需要显式使用：
   ```bash
   zig build -Dexamples=true
   ```
2. XML 动态 writer 的安全使用约束更严格：
   - 当输出目标来自 `std.Io.Writer.Allocating` 时，不能按值传递 `allocating.writer`
   - 应使用 `zzig.xml.createAllocatingWriter(...)`，或显式传 `&allocating.writer`
3. 文件路径 API 新增：
   - `zzig.File.CurrentPath()` 继续返回 `[]u8`
   - `zzig.File.CurrentPathZ()` 新增返回 `[:0]u8`

上述变化均已补充到 README 与迁移文档中。

### 1.2 性能结论

当前阶段已确认两类结论：

1. **已修复的兼容问题本身不会带来语义回退**，尤其是：
   - `std.Io.Writer.Allocating.writer` 不能按值拷贝
   - `fromArrayList(...)` 写入后必须在读取前同步 `toArrayList()`
2. **已落地的一处低风险性能优化**：
   - `RandomString()` 已改为在单次调用内复用随机源，而不是每生成一个字符就重建一次 `IoSource` / `Random`
3. **已落地的一处增量性能 API**：
   - 新增 `zzig.xml.writeToFileStreaming(...)`，用于降低大 XML 文档写文件时的峰值内存占用，同时保留原 `writeToFile()` 语义不变
4. **已落地的一处解析热路径优化**：
   - `Reader` 与 `DOM` 在属性值/文本不含实体引用时，直接走复制或追加分支，跳过 `decodeText + Allocating writer` 的中间开销
5. **已落地的一处文本聚合优化**：
   - `Reader.readElementTextAlloc()` 直接返回局部聚合缓冲的 `owned slice`，去掉返回前的最终重复复制
6. **已落地的一处实体匹配优化**：
   - `scanner.decodeText()` 将命名实体匹配从串行多次 `std.mem.eql` 调整为按长度和首字符分支，减少常见实体解码时的比较开销

## 3. 已落地优化

### 3.1 RandomString 随机源复用

文件：`src/random/randoms.zig`

#### 优化前
- 每生成一个字符都会重新构造一次 `std.Random.IoSource`
- 每生成一个字符都会重新获取一次 `Random` 接口

#### 优化后
- 在 `RandomString()` 单次调用内只初始化一次随机源
- 整个字符串生成过程复用同一个 `std.Random`

#### 影响
- **语义不变**
- **固定开销下降**
- 对长字符串生成路径更有利

### 3.2 XML 流式文件写出 API

文件：
- `src/xml/dom.zig`
- `src/xml/xml.zig`

#### 优化前
- `writeToFile()` 路径会先把完整 XML 文档拼到内存，再整块写入文件

#### 优化后
- 保留原 `writeToFile()` 语义不变
- 新增 `writeToFileStreaming()`，通过小缓冲直接流式写入文件

#### 影响
- **现有 API 语义不变**
- **为大文档输出新增更低峰值内存的写出路径**
- 流式写出若中途失败，目标文件可能已包含部分内容，因此作为增量 API 提供而非替换旧接口

### 3.3 XML 无实体快速路径

文件：
- `src/xml/reader.zig`
- `src/xml/dom.zig`

#### 优化前
- 即使属性值或文本里完全不含 `&`，也会进入 `decodeText(...)`
- 这条路径会创建 `std.Io.Writer.Allocating` 并经过中间缓冲回写

#### 优化后
- 对不含实体引用的常见输入先做一次 `&` 检查
- 未命中实体时，直接 `dupe` 或 `appendSlice`
- 只有真正包含实体时，才进入原有解码流程

#### 影响
- **语义不变**
- **普通 XML 文本和属性解析的固定开销下降**
- **实体解码分支保持原实现，不改变兼容行为**

### 3.4 Reader 文本聚合返回优化

文件：
- `src/xml/reader.zig`

#### 优化前
- `readElementTextAlloc()` 使用局部 `ArrayList` 聚合文本
- 返回前再执行一次 `dupe(...)`，造成额外分配和拷贝

#### 优化后
- 保持局部聚合逻辑不变
- 结束时直接 `toOwnedSlice()` 转移所有权返回

#### 影响
- **语义不变**
- **减少一次最终分配与复制**
- 对较大元素文本聚合更有利

### 3.5 Scanner 实体匹配分支压缩

文件：
- `src/xml/scanner.zig`

#### 优化前
- 每次命名实体解码都会顺序尝试多次 `std.mem.eql`
- 常见的 `amp/lt/gt/apos/quot` 需要经过重复比较才能命中

#### 优化后
- 先处理数字实体路径
- 命名实体按长度和首字符分支匹配
- 未知实体仍保持原样输出

#### 影响
- **语义不变**
- **减少常见命名实体的比较次数**
- **数字实体与未知实体行为保持原实现**

## 4. 现有性能热点分析

以下热点已识别，但本轮未直接改写，以避免在“性能提升”过程中引入新的语义边界。

### 4.1 XML 默认写文件接口仍走整块缓冲

涉及文件：
- `src/xml/writer.zig`
- `src/xml/dom.zig`

当前实现是：
1. 先把完整 XML 序列化到内存缓冲区
2. 再整块写入文件

#### 优点
- 语义边界清晰
- 写入失败前不会对目标文件产生部分内容

#### 成本
- 峰值内存与输出大小线性相关
- 对大 XML 文档存在明显的中间缓冲开销

#### 当前处理
- 已新增 `writeToFileStreaming()` 作为低内存写出路径
- 旧 `writeToFile()` 保留，用于需要维持原有语义边界的调用方

### 4.2 DOM 解析的实体解码存在临时缓冲复制

涉及文件：
- `src/xml/dom.zig`
- `src/xml/reader.zig`

当前属性值/文本解码过程通常是：
1. 先写入 scratch 或临时 `ArrayList`
2. 再复制到最终返回值或 DOM 结构

#### 影响
- 对大量带实体引用的属性和文本会产生额外复制
- 不是 correctness 问题，但会增加解析成本

#### 备注
这是下一阶段可以继续优化的方向，但需要仔细处理所有权与 ArenaAllocator 生命周期。

### 4.3 CurrentPath 的兼容税

涉及文件：
- `src/file/file.zig`

当前 `CurrentPath()` 为保持旧语义返回 `[]u8`，会：
1. 调用 `CurrentPathZ()` 获取 `[:0]u8`
2. 再复制为普通切片

#### 影响
- 对高频调用路径存在一次额外分配与复制

#### 当前处理
- 已新增 `CurrentPathZ()`
- 若调用方能接受 sentinel 路径，可主动迁移到 `CurrentPathZ()` 以避免这部分成本

## 5. 基准运行结果

本次实际运行了现有 JSON 基准：

```bash
zig build json-bench -Dexamples=true
```

终端中文输出在当前 Windows 编码环境下存在乱码，但**数值结果可正常读取**。

### 5.1 小型 JSON（约 261 字节）
- 原版解析器：20.58 μs
- 零分配优化：5.01 μs
- 紧凑格式：19.88 μs
- 零分配相对原版：**4.11x**

### 5.2 中型 JSON（约 5235 字节）
- 原版解析器：285.22 μs
- 零分配优化：81.25 μs
- 紧凑格式：318.80 μs
- 零分配相对原版：**3.51x**
- 堆分配次数下降：**100%**

### 5.3 大型 JSON（约 64421 字节）
- 原版解析器：2777.74 μs
- 零分配优化：1153.18 μs
- 紧凑格式：3120.11 μs
- 零分配相对原版：**2.41x**
- 堆分配次数下降：**100%**

### 5.4 结论
- 当前 JSON 零分配路径保持明显性能优势
- 本次 Zig 0.16 兼容与收尾改动未破坏现有 JSON 基准能力
- 当前最值得继续优化的性能点不在 JSON，而在 XML 文件写入和 XML 解码中间缓冲

## 6. 后续建议

### 优先级 1
为 Reader / DOM 解码路径补“调用方提供输出缓冲”或“复用 scratch 视图”的接口，减少属性值与文本的二次复制。

### 优先级 2
对 `CurrentPath()` 的实际调用点做审计，将高频路径迁到 `CurrentPathZ()`。
